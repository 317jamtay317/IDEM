## 1. TDD is non-negotiable — Red → Green → Refactor

Every change to production code starts with a failing test.

**The loop, every time:**
1. **Red** — write the smallest test that captures the next behavior. Run it. **It must fail for the right reason** (assertion failure or missing API — not a typo, missing using, or compile error in unrelated code).
2. **Green** — write the *minimum* production code that makes the test pass. No extra cases, no premature abstraction.
3. **Refactor** — with the test green, clean up names, extract methods, collapse duplication. Re-run tests after each refactor step.
4. Commit only when green.

**Forbidden:**
- Writing production code before a failing test exists for it.
- Disabling, skipping (`Skip = "..."`), or weakening a test to make the build pass. If a test is wrong, fix the test in its own commit with a written justification.
- "I'll add the tests after" — there is no "after."

**Allowed without a prior test:** scaffolding (project files, csproj, DI registration boilerplate, migrations generated by EF), pure formatting, comment-only edits, and deleting dead code that has no test coverage to begin with.

---

## 2. Verify coverage before every commit

Architecture.md mandates **≥ 80% code coverage**, enforced in CI. Don't let CI be the first thing to check.

**Before `git commit` on any branch:**
1. `dotnet test` — all tests pass.
2. `dotnet test --collect:"XPlat Code Coverage"` (or the configured coverage command) — overall line coverage ≥ 80%.
3. If coverage dropped on the lines you touched, add the missing tests **before** committing — not in a follow-up.
4. State the numbers in the commit body: `Tests: 142 passed. Coverage: 84.3% (was 84.1%).`

If you cannot run coverage locally (tooling missing, etc.), say so explicitly in chat and stop — do not commit blind.

## 3. Clean Architecture — dependencies point inward

Layout (see Architecture.md):

```
Api ─── Application ─── Domain
  └─── Infrastructure ──┘
```

- **Domain** depends on nothing. No EF, no Wolverine, no ASP.NET, no `Microsoft.Extensions.*`. Pure C#.
- **Application** depends on Domain only. Defines `interface`s for infra needs (repositories, gateways, clock, etc.).
- **Infrastructure** depends on Domain + Application. Implements the interfaces. EF Core, Wolverine outbox, SQL Server, external HTTP all live here.
- **Api** depends on Application + Infrastructure (the latter only at the composition root, `Program.cs`).

If a `using RecordKeeping.Infrastructure...` appears anywhere in Domain or Application, that's a bug — fix the design, not the symptom.

## 4. Domain-Driven Design — use the ubiquitous language

`UbiquitousLanguage.md` is the only allowed vocabulary for domain types, properties, parameters, test names, and user-visible strings.

- **Tenant**, not Company / Account / Organization / Customer.
- **Facility**, not Plant / Site / Location (the entity is `Facility`; "plant" is fine in customer-facing copy where the term is `Asphalt Plant`).
- **Record**, **Report**, **Report Template**, **Regulator**, **MDEQ**, **IDEM** — spelled exactly as written.
- **Rieth-Riley** — hyphenated, R-I-E-T-H, everywhere.

If a needed term is not in `UbiquitousLanguage.md`, add it there first (with `🟡` if tentative), then write the code.

**Aggregates own their invariants.** Business rules live on entities/aggregates, not in handlers or services. Handlers orchestrate; they do not decide.

## 5. Invariants are traceable

Every invariant in [Invariants.md](docs/Invariants.md) has a stable ID `I-D##`. When you enforce one, tag it:

- **Domain code**: comment near the check — `// I-D01: Records cannot exist without a TenantId.`
- **FluentValidation**: `.WithErrorCode("I-D03")`.
- **Tests**: trait or category — `[Trait("Invariant", "I-D01")]` (or the xUnit v3 equivalent in use). Every invariant must have **at least one** test that proves it holds, plus negative tests where applicable.

Invariant IDs are **append-only** — never renumber. Retiring an invariant keeps its ID reserved with a `Retired` note.

---

## 6. Results, not exceptions, for business outcomes

- Use `ErrorOr<T>` (per Architecture.md) for any operation whose failure is a *business outcome* (validation failed, aggregate not found, permission denied, invariant violated).
- Throw exceptions only for *genuinely unexpected* failures (DB unreachable, serialization broken, programmer error).
- Never use exceptions for control flow.

---

## 7. Documentation — public and protected only

Per the user's global rule and Architecture.md:

- **Every** `public` and `protected` type and member has an XML doc comment (`/// <summary>...</summary>`). No exceptions.
- `internal` and `private` are exempt — but a one-line `//` comment on non-obvious internals is welcome.
- Doc comments describe *intent and contract*, not the implementation. "Returns the Tenant for the given id, or `Errors.Tenant.NotFound` if no such Tenant exists" — not "calls the repository."

---

## 8. Tenancy is enforced at every layer

**I-D03** says no query, report, export, or background job may return another Tenant's data. This is a security invariant, not a feature.

- All Tenant-scoped reads filter by `TenantId` at the query level (preferably via a global EF query filter that cannot be forgotten).
- Every read endpoint has a negative test: "request as Tenant A, assert no Tenant B data is returned." This is mandatory, not optional.
- A violation of tenant isolation is a security incident — treat any PR review finding here as a blocker.

## 9. Git workflow

- Never commit directly to `main`. Branch first (`claude/<descriptor>` or feature-named).
- Commit messages: short imperative subject ≤ 72 chars, body explaining *why*. Include the test+coverage line from §2.
- End every commit message with the trailer the harness sets automatically (`Co-Authored-By: Claude Opus ...`).
- Never use `--no-verify`, `--no-gpg-sign`, or skip hooks unless explicitly asked.
- Don't push or open PRs unless the user asks.
